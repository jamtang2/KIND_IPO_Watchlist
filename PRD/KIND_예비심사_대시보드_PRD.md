# KIND 상장예비심사 현황 대시보드 PRD

## 0. 문서 정보
- 작성 목적: Claude Code에게 전달하여 자동화 대시보드를 구현시키기 위한 제품 요구사항 정의서(PRD)
- 프로젝트명(제안): `KIND_PreListing_Dashboard`
- 배포 방식(제안): GitHub Actions(스케줄 수집) + GitHub Pages(정적 대시보드), 기존 CapEx_Dashboard / Shortage_Bottleneck_Scanner와 동일한 패턴

---

## 1. 개요 및 목적

한국거래소 KIND 사이트의 **IPO현황 > 예비심사기업** 페이지에서 "청구서 접수" 상태인 기업을 추적하여, 심사 경과일과 심사 결과(승인/미승인/철회)를 한눈에 볼 수 있는 대시보드를 제작한다. 심사 결과가 확정된 기업은 30일간 리스트에 노출 후 자동 제거되는 롤링(rolling) 리스트 형태로 운영한다.

**핵심 소스 페이지**
```
https://kind.krx.co.kr/listinvstg/listinvstgcom.do?method=searchListInvstgCorpMain
```

이 페이지는 검색 조건을 서버에 AJAX로 전송하여 결과 테이블을 비동기로 렌더링하는 구조이므로, 초기 HTML 응답만으로는 데이터를 얻을 수 없다. Milestone 1에서 이 부분을 최우선으로 규명해야 한다(자세한 내용은 4장 참고).

## 2. 필터 조건 (요구사항 2번 반영)

KIND 예비심사기업 페이지에서 확인된 실제 검색 옵션은 다음과 같다.

| 필터 | 옵션 | 대시보드 적용 값 |
|---|---|---|
| 시장구분 | 전체 / 유가증권 / 코스닥 | **전체** |
| 상장유형 (다중선택) | 신규상장 / 이전상장 / 재상장 / 스팩존속합병 / 스팩소멸합병 | **신규상장, 이전상장만 체크** |
| 심사결과 (다중선택) | 청구서접수 / 심사승인 / 심사미승인 / 심사철회 / 공모철회 / 상장철회 / 승인효력기간만료 / 상장승인 | 전체 조회 후 파이프라인에서 재분류(3장 참고) |
| 청구일 | 기간(From~To) | 최근 N개월(예: 최근 12개월, 설정 가능하게) — 청구서 접수 후 30일 넘게 미해결인 장기 건도 놓치지 않도록 넉넉히 설정 |

> 참고: "공모철회", "상장철회", "승인효력기간 만료" 등도 존재하지만 요구사항 5번에서는 심사승인/심사미승인/심사철회 3종류만 명시했으므로, 나머지 상태는 별도 처리 규칙을 3.3에 정의한다.

## 3. 핵심 비즈니스 로직

### 3.1 신규 등록 로직 (요구사항 3, 4번)
- 심사결과가 **"청구서 접수"** 상태인 기업이 KIND에 새로 나타나면 대시보드 리스트에 **신규 등록**한다.
- 청구서 접수일(청구일)을 기준으로 **경과일(D+n)**을 계산한다.
  - `경과일 = 오늘 날짜 - 청구서 접수일` (영업일이 아닌 일반 캘린더일 기준을 기본값으로 하되, 영업일 기준 옵션도 설정 가능하게 구현)
  - **계산 시점 주의**: 데이터 수집(및 상태 diff 판정)은 주 1회(목요일 21:00 KST)만 이루어지지만, "경과일" 숫자 자체은 프론트엔드에서 접속 시점의 실제 날짜 기준으로 클라이언트 사이드에서 재계산하여 표시한다. 즉 JSON에는 `apply_date`(청구일)만 저장하고, 화면에 뿌릴 때 `오늘 - apply_date`를 매번 계산한다. 이렇게 해야 목요일 사이에 접속해도 경과일이 실제와 어긋나지 않는다. (30일 만료 판정은 데이터가 갱신되는 목요일 배치 시점에만 이루어지므로 최대 6일의 지연이 있을 수 있음 — 10장 리스크에 명시)

### 3.2 심사 현황 컬럼 및 색상 규칙 (요구사항 5번)

| 심사결과(KIND 원본값) | 대시보드 표시 | 색상 |
|---|---|---|
| 청구서 접수 | 진행중 | 회색/파랑(기본, 강조 없음) |
| 심사승인 | 심사승인 | 🟢 녹색 |
| 심사미승인 | 심사미승인 | 🔴 적색 |
| 심사철회 | 심사철회 | 🟡 황색 |
| 공모철회 / 상장철회 / 승인효력기간만료 | 해당 값 그대로 표시 | ⚪ 회색(중립, 확정 상태로 간주하여 3.3 만료 로직 적용) |

### 3.3 30일 만료(자동 삭제) 로직 (요구사항 6번)
- 심사결과가 "청구서 접수"에서 **확정 상태**(심사승인/심사미승인/심사철회, 및 위 기타 확정성 상태 포함)로 **최초 변경된 날짜**를 `result_date`로 기록한다.
- `오늘 날짜 - result_date > 30일` 이 되는 시점에 해당 기업을 리스트에서 **삭제**한다(단, 원본 이력 로그에는 보관 — 6장 참고).
- "진행중" 상태인 기업은 만료 로직이 적용되지 않고 무기한 유지된다.

### 3.4 상태 전이(state transition) 요약
```
[신규 발견: 청구서 접수] → 리스트 등록, 상태='진행중', result_date=null
        │  (매일 재조회)
        ▼
상태 변경 감지: 청구서 접수 → 심사승인/심사미승인/심사철회/기타확정
        │
        ▼
result_date = 변경 감지일, 상태 컬럼 업데이트 + 색상 적용
        │  (매일 경과일 체크)
        ▼
(오늘 - result_date) > 30일 → 리스트에서 제외 (아카이브로 이동)
```

## 4. 데이터 수집 전략

### 4.1 수집 방식 우선순위 (Claude Code가 순서대로 시도)

**1순위 — EXCEL 다운로드 엔드포인트 활용 (완전 자동화)**
KIND 예비심사기업 페이지에는 검색 조건과 함께 결과를 엑셀로 내려받는 "EXCEL" 버튼이 있다. 이 버튼이 브라우저에서 클릭될 때 실제로 어떤 HTTP 요청(URL, method, form data)이 나가는지를 **개발 단계에서 딱 한 번** 브라우저 개발자도구(Network 탭)로 캡처한다. 이후 그 요청을 Python `requests` 세션 코드로 그대로 재현하도록 스크립트를 작성하면, 매주 GitHub Actions가 사람 개입 없이 자동으로 동일한 요청을 보내고 파일을 받아온다. **즉 사용자가 매번 수동으로 엑셀을 다운로드할 필요는 전혀 없으며, Network 탭 캡처도 최초 개발 시 1회만 필요하다.** 이 방식이 성공하면 HTML 파싱보다 훨씬 안정적이다.

**2순위 — 검색 결과 AJAX 엔드포인트 직접 호출**
"검색" 버튼 클릭 시 호출되는 AJAX 요청(대개 `method=searchListInvstgCorpSub` 유사 패턴, KRX 계열 사이트에서 흔한 구조)을 Network 탭에서 캡처하여 JSON 또는 HTML fragment 응답을 파싱한다.

**3순위 — 헤드리스 브라우저(Playwright) 렌더링 후 스크래핑**
위 두 방식이 세션/토큰 검증 등으로 막힐 경우, Playwright로 실제 페이지를 열고 필터를 설정한 뒤 렌더링된 테이블을 파싱한다. GitHub Actions에서 실행 가능하도록 headless Chromium 설치 스텝 포함.

> **Claude Code 작업 지시**: Milestone 1 착수 시, 반드시 실제 브라우저에서 Network 탭을 열고 "검색" 및 "EXCEL" 클릭 시의 실제 요청을 캡처하는 것부터 시작할 것. 위 URL 패턴은 추정이며 확정된 스펙이 아니다.

### 4.2 수집 파라미터 매핑(예상)
| 폼 항목 | 예상 파라미터명(확인 필요) | 값 |
|---|---|---|
| 시장구분 | `marketType` 등 | 전체(공백 또는 '0') |
| 상장유형 | `listType[]` 등 | 신규상장, 이전상장에 해당하는 코드값 |
| 심사결과 | `examResult[]` 등 | 전체 선택(모든 상태 값 체크) 후 파이프라인에서 재분류 |
| 청구일 시작/종료 | `fromDate`, `toDate` 등 | 최근 N개월 |

실제 파라미터명은 Milestone 1에서 확정하여 이 표를 업데이트한다.

### 4.3 수집 주기
- GitHub Actions cron, **매주 목요일 밤 9시(21:00 KST) 1회** 실행
- cron 표현식은 UTC 기준으로 등록해야 하므로 변환 필요: KST 21:00 = UTC 12:00(당일, -9시간) → `cron: '0 12 * * 4'` (UTC 목요일 12:00 = KST 목요일 21:00)
- **60일 비활성 정책 주의**: GitHub Actions는 저장소에 60일간 커밋/활동이 없으면 스케줄 워크플로우를 자동 비활성화한다. 주 1회 실행(월 4~5회)은 이 임계값보다 훨씬 잦으므로 문제없으나, 워크플로우 파일에 안내 주석을 남겨둔다.
- **직접 실행(workflow_dispatch) 지원 필수**: 목요일 실행을 놓쳤거나 특정 시점 데이터를 즉시 확인하고 싶을 때 수동으로도 실행 가능하도록 설정

## 5. 데이터 모델 (JSON, 이력 관리 포함)

```json
{
  "last_updated": "2026-07-01T18:00:00+09:00",
  "companies": [
    {
      "corp_name": "예시기업",
      "market": "코스닥",
      "listing_type": "신규상장",
      "apply_date": "2026-06-01",
      "status_raw": "심사승인",
      "status_display": "심사승인",
      "status_color": "green",
      "elapsed_days_at_apply": 30,
      "result_date": "2026-07-01",
      "days_since_result": 0,
      "expire_date": "2026-07-31",
      "history": [
        {"date": "2026-06-01", "status": "청구서 접수"},
        {"date": "2026-07-01", "status": "심사승인"}
      ]
    }
  ],
  "archived": [
    { "...": "만료되어 리스트에서 제외된 기업 (감사/이력 목적 보관)" }
  ]
}
```

- `history` 배열을 통해 상태 변경 이력을 모두 보존 (CapEx_Dashboard의 revision history 방식과 동일한 설계 원칙)
- 삭제 대신 `archived`로 이동시켜 원본 데이터 유실 방지

## 6. 파이프라인 (주간 실행 단계, 매주 목요일 21:00 KST)

1. **필터 적용 조회**: 시장구분=전체, 상장유형=[신규상장, 이전상장], 청구일 범위=최근 N개월로 KIND 조회
2. **원본 데이터 정규화**: 회사명/시장/상장유형/청구일/심사결과를 표준 스키마로 매핑
3. **기존 저장 데이터와 비교(diff)**:
   - 신규 "청구서 접수" 기업 → `companies`에 신규 추가
   - 기존 기업의 상태가 변경됨 → `history`에 추가, `status_raw`/`result_date` 갱신
   - 상태 변경 없음 → `elapsed_days` 등 파생값만 갱신
4. **경과일 계산**: `elapsed_days_at_apply`(진행중 기업), `days_since_result`(확정 기업) 재계산
5. **만료 처리**: `days_since_result > 30`인 기업을 `companies`에서 `archived`로 이동
6. **색상/표시값 매핑**: 3.2 규칙 적용
7. **JSON 저장 및 커밋**: 정적 사이트가 참조하는 JSON 파일 업데이트 → git commit/push → GitHub Pages 자동 반영

## 7. 대시보드 UI 요구사항

- **메인 리스트 뷰** (테이블)
  - 컬럼: 회사명, 시장구분, 상장유형, 청구일, 경과일(D+n), 심사현황(색상 배지), 최종갱신일
  - 심사현황 배지: 진행중(중립), 심사승인(🟢), 심사미승인(🔴), 심사철회(🟡)
  - 기본 정렬: 경과일 내림차순(오래 대기 중인 기업이 상단)
- **필터/토글**
  - 시장구분(전체/유가증권/코스닥) 재필터링
  - 상장유형(신규상장/이전상장) 재필터링
  - "진행중만 보기" / "확정결과 포함 보기" 토글
- **요약 카드**: 진행중 기업 수, 이번 주 신규 접수, 이번 주 결과 확정(승인/미승인/철회 각각)
- **경과일 강조**: 예를 들어 45영업일(코스닥 표준 심사기간 참고) 초과 시 시각적 강조(예: 굵게, 경고 아이콘) — 선택 기능으로 제안, 확정 요구사항은 아님

## 8. 기술 스택 및 배포 (제안, 기존 프로젝트와 동일 패턴)

- 저장: JSON 파일 (repo 내 커밋 방식, 기존 프로젝트와 동일)

- 스케줄링: GitHub Actions cron (KST 변환 필수), 60일 비활성 정책 고려

## 10. 리스크 및 제약사항
- KIND 페이지가 AJAX 기반이라 **정확한 요청 스펙 확인 전에는 구현 불가** — M1이 전체 프로젝트의 선행 조건
- KIND 사이트는 봇 차단(User-Agent 검증, 세션/쿠키 필요 등) 정책이 있을 수 있으므로 요청 헤더 구성에 유의
